# Generation runs as a background job; the frontend polls for readiness

Producing a Directed Drawing — LLM structured generation of the whole Primitive set, schema validation, then pre-generating per-Step Narration audio (ADR-0007) — takes seconds to tens of seconds, too long for a blocking web request. On Plan confirmation the app enqueues a background job (Solid Queue, the Rails 8 default — no Redis). The job generates, validates against the DSL schema, retries on malformed output, synthesizes narration, stores audio via Active Storage, and marks the Directed Drawing ready.

Because the stack is Inertia + React (not Hotwire/Turbo), the preview/loading screen polls a status endpoint (Inertia polling) showing a friendly "Drawing your picture…" state until the job completes, then reveals the confirmation preview. We chose polling over Action Cable websockets: this is a one-shot, fire-and-wait generation, so websockets are overkill and polling is simpler and sufficient. The surrounding stack uses Rails defaults — native Rails 8 authentication (email/password sessions, no Devise), Solid Queue over Sidekiq/Redis, Active Storage for both Artwork photos and Narration audio, backed by Cloudflare R2 (S3-compatible) in all cloud environments — chosen for minimal moving parts and zero egress fees on served audio/images. R2's S3 compatibility keeps the storage backend swappable.
